## Epic 1: 用户管理

### Feature 1.1: 用户登录与注册

#### Story 1.1.1: 账号密码登录

**用户角色**: 平台用户（普通员工、管理员、SKILL开发者）
**用户需求**: 使用公司内部账号密码登录平台，访问SKILL资源
**业务价值**: 确保只有授权用户可以访问平台，保障资源安全

**验收条件**:

**正常场景**:
```
Given 用户已注册公司内部账号
When 用户输入正确的工号和密码，点击登录
Then 系统验证通过，跳转至平台首页，显示用户个人信息
```

**异常场景**:
```
Given 用户已注册公司内部账号
When 用户输入错误的密码
Then 系统提示"密码错误，请重新输入"，并显示剩余尝试次数

Given 用户已注册公司内部账号
When 用户连续5次输入错误密码
Then 系统锁定账号，提示"账号已锁定，请联系管理员解锁"
```

**交接标注**: 已明确

---

#### Story 1.1.2: 单点登录（SSO）

**用户角色**: 平台用户
**用户需求**: 通过公司统一身份认证系统登录，无需重复输入密码
**业务价值**: 简化登录流程，提升用户体验，与公司现有系统集成

**验收条件**:

**正常场景**:
```
Given 用户已在公司其他系统登录
When 用户点击"SSO登录"按钮
Then 系统跳转至公司统一认证页面，认证通过后自动登录平台
```

**异常场景**:
```
Given 用户未在公司其他系统登录
When 用户点击"SSO登录"按钮
Then 系统跳转至公司统一认证页面，用户输入凭证后完成登录
```

**交接标注**: 重点细化 - 需确认SSO协议（OAuth2/OIDC）和公司认证系统对接方式

---

#### Story 1.1.3: 用户注册

**用户角色**: 公司内部员工
**用户需求**: 注册平台账号，填写个人信息
**业务价值**: 确保用户身份真实，便于平台管理

**验收条件**:

**正常场景**:
```
Given 员工未注册过平台账号
When 员工填写工号、姓名、部门、邮箱等信息，提交注册申请
Then 系统提示"注册申请已提交，等待管理员审核"，并发送邮件通知
```

**异常场景**:
```
Given 员工已注册过平台账号
When 员工再次提交注册申请
Then 系统提示"该工号已注册，请直接登录"
```

**交接标注**: 已明确

---

#### Story 1.1.4: 密码管理

**用户角色**: 平台用户
**用户需求**: 修改密码、重置忘记的密码
**业务价值**: 保障账号安全，方便用户自主管理密码

**验收条件**:

**正常场景**:
```
Given 用户已登录平台
When 用户输入旧密码、新密码、确认新密码，点击修改
Then 系统验证旧密码正确，新密码符合复杂度要求，提示"密码修改成功"

Given 用户忘记密码
When 用户点击"忘记密码"，输入工号，系统发送验证码至注册邮箱
Then 用户输入验证码后，可设置新密码
```

**异常场景**:
```
Given 用户已登录平台
When 用户输入的旧密码错误
Then 系统提示"旧密码错误，请重新输入"

Given 用户设置新密码
When 新密码不符合复杂度要求（8位以上、包含字母、数字、特殊符号）
Then 系统提示"密码复杂度不足，请包含字母、数字和特殊符号"
```

**交接标注**: 已明确

### Feature 1.2: 权限管理

#### Story 1.2.1: 角色权限区分

**用户角色**: 超级管理员
**用户需求**: 为不同用户分配不同角色，控制其可访问的功能和资源
**业务价值**: 确保资源安全，防止未授权访问

**验收条件**:

**正常场景**:
```
Given 超级管理员已登录
When 管理员将用户角色从"普通用户"修改为"SKILL开发者"
Then 用户获得提交公共SKILL、参与套件SKILL维护等额外权限
```

**异常场景**:
```
Given 普通用户已登录
When 普通用户尝试访问管理员功能（如用户管理、SKILL审核）
Then 系统提示"权限不足，无法访问该功能"
```

**交接标注**: 重点细化 - 需明确四种角色（超级管理员、普通管理员、普通用户、SKILL开发者）的完整权限列表

#### Story 1.2.2: SKILL访问权限控制

**用户角色**: SKILL开发者/管理员
**用户需求**: 为单个SKILL设置访问权限（个人可见、部门可见、全公司可见）
**业务价值**: 精准控制SKILL资源的可见范围，保护敏感资源

**验收条件**:

**正常场景**:
```
Given 开发者上传了一个SKILL
When 开发者将SKILL可见范围设置为"部门可见"
Then 只有同部门用户可以查看和下载该SKILL
```

**异常场景**:
```
Given 某SKILL的可见范围设置为"部门A可见"
When 部门B的用户尝试访问该SKILL
Then 系统提示"您无权访问该SKILL"
```

**交接标注**: 已明确

#### Story 2.1.1: 提交公共SKILL审核

**用户角色**: SKILL开发者
**用户需求**: 将个人SKILL提交至公共广场，供全公司用户使用
**业务价值**: 促进SKILL资源共享，提升复用率

**验收条件**:

**正常场景**:
```
Given 开发者已上传个人SKILL
When 开发者填写SKILL信息（名称、描述、版本、分类、适用场景），提交审核

```

**异常场景**:
```
Given 开发者提交SKILL审核

Then 系统提示校验失败原因，开发者可修改后重新提交

Given 开发者提交SKILL审核
When 管理员审核失败
Then 系统将失败原因反馈给开发者，开发者可修改后重新提交
```

**交接标注**: 已明确
